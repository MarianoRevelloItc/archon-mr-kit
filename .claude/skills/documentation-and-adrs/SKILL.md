# Documentation and ADRs

The kit prefers **decisions captured at decision time** over docs reverse-engineered later. Three forms:

1. **Architecture Decision Records (ADRs)** — the *why* behind structural choices. Nygard format.
2. **API documentation** — the *why-not-the-what* of every public function/route.
3. **READMEs** — hand-curated, never autogenerated. The first thing a new teammate reads.

---

## Form 1: ADRs (Architecture Decision Records)

An ADR captures **a single architectural decision** at the moment it's made. Format: Michael Nygard, 2011.

### Canonical template

```markdown
# {NUMBER}. {Title — present tense, declarative: "Use Postgres for relational data"}

Date: {YYYY-MM-DD}

## Status

{Proposed | Accepted | Deprecated | Superseded by ADR-NNNN}

## Context

{Forces in play. Constraints. What problem we're solving. Avoid solutions here — just describe
the situation. 1-3 paragraphs. If this section reads like a Wikipedia article, you're padding.}

## Decision

{The change we're making. Active voice, present tense: "We will use X." 1 paragraph.}

## Consequences

{Both positive and negative. What becomes easier? What becomes harder? What new problems
does this create? Be honest — sweeping consequences under the rug makes the ADR useless
when someone six months from now asks "why did we do this?"}
```

### When to write an ADR

Write one when **a future engineer will need to know why this decision was made**:

- Choosing a database engine, a framework, a language for a service.
- Choosing between two API styles (REST vs GraphQL, sync vs async).
- Choosing a deployment topology (monorepo vs polyrepo, monolith vs services).
- Adopting or dropping a major dependency.
- Defining a cross-cutting convention (error handling, logging, auth).

Don't write ADRs for: tactical choices (which lodash function to use), preferences (tabs vs spaces — put in CLAUDE.md), or things already settled by the framework you chose.

### Numbering and lifecycle

- ADRs are **immutable once accepted**. If a decision changes, write a new ADR with status "Accepted" and update the old one's status to "Superseded by ADR-NNNN".
- Number them sequentially: `0001-record-architecture-decisions.md`, `0002-...`. Zero-pad to 4 digits.
- Live in `docs/adr/`. Created by `mr-bootstrap-project` Phase 8b.

### Worked example 1: "We chose JWT over OAuth for our auth"

```markdown
# 0007. Use JWT (not OAuth) for first-party API auth

Date: 2026-04-15

## Status

Accepted

## Context

We need authentication for our first-party SPA hitting our own backend. Options considered:

- **OAuth 2.0 + identity provider (Auth0, Clerk)**: industry standard, handles flows, costs $X/MAU.
- **Session cookies**: simple, server-side state, requires sticky sessions or shared session store.
- **JWT (HS256, server-issued)**: stateless, easy to scale, no third-party dependency.

We have one frontend, one backend, no third-party identity providers, and < 10K users at this stage.
Cost matters, third-party dependency adds an outage surface, and we control both ends of the auth
handshake.

## Decision

We will use JWT (HS256-signed, server-issued, 1-hour access tokens, 30-day refresh tokens stored
in HttpOnly Secure SameSite=Strict cookies). The signing key is in the secrets manager, rotated
quarterly. Token verification happens in a FastAPI dependency injected on every protected route.

## Consequences

**Positive:**
- No third-party dependency for auth → one less outage surface.
- $0 marginal cost up to ~1M users.
- Simple to debug locally (no OAuth dance).

**Negative:**
- We own all the security work: rate-limiting, password hashing, CSRF, refresh-token rotation. The
  `security-and-hardening` skill is mandatory on every PR that touches auth.
- Migrating to OAuth later means a coordinated frontend+backend cutover. We accept this if and when.
- Token revocation is harder (we can't blacklist a JWT without a server-side check, defeating
  statelessness). We mitigate via short access-token TTLs.
```

### Worked example 2: "Vertical slicing over feature branches"

```markdown
# 0012. Use vertical slices, not feature branches, for issue work

Date: 2026-04-22

## Status

Accepted

## Context

The team has tried two patterns for issue → PR work:

- **Feature branches**: long-lived branch, big PR, late merge, lots of conflicts.
- **Vertical slices**: each issue is a thin end-to-end change, merged within a day, behind a feature flag if incomplete.

Our pilot ran 8 issues with each pattern. Feature branches averaged 4.2 days to merge with
2.8 conflicts each. Vertical slices averaged 0.9 days to merge with 0.3 conflicts each.

The kit's `mr-issue-to-pr` workflow is built around the vertical-slice assumption.

## Decision

We will scope every issue as a vertical slice: one acceptance criterion → one PR → mergeable
within a day. Multi-AC features are split into N issues by `mr-prd-to-issues`. Incomplete UX
hides behind feature flags.

## Consequences

**Positive:**
- `git bisect` becomes trivial (each commit is small, atomic).
- Code review fits in 30 minutes per PR.
- Production rollback is per-slice, not per-feature.

**Negative:**
- More PRs to track. We mitigate with the `auto-followup` and `prd` labels.
- Feature flags accumulate. We add a quarterly chore to retire them (issue: kit-todo).
- Issues sometimes feel "incomplete" alone. Reviewers must read the parent PRD for context — we
  enforce this by linking `Fixes #N` (where N is the PRD issue) on every child PR.
```

## Form 2: API documentation

Document **why, not what**. The signature shows what; the docstring explains why.

```python
# BAD — restates the signature in English
def calculate_score(company: Company, signals: list[Signal]) -> float:
    """Calculate the score for a company given a list of signals.

    Args:
        company: The company.
        signals: The signals.

    Returns:
        The score.
    """

# GOOD — explains the why and the gotchas
def calculate_score(company: Company, signals: list[Signal]) -> float:
    """Composite reachability score in [0.0, 1.0].

    The score weights signals (executives, website, recent press releases) per the
    rubric in ADR-0014. Returns 0.0 (not None) for companies with no signals so
    downstream sorts are stable. Caller is responsible for caching — this fn is
    pure but Signal.is_recent() can be slow on cold cache.
    """
```

For HTTP routes:

```python
@app.post("/api/v1/companies/{cid}/score")
async def post_score(cid: int, body: ScoreRequest) -> ScoreResponse:
    """Recompute a company's reachability score.

    Why: reachability is cached on Company.score_cached_at; clients trigger
    recomputation when they suspect staleness (new signals just landed). Idempotent
    within ~1 second — repeated calls in a tight window return the cached value.

    See ADR-0014 for the scoring rubric.
    """
```

## Form 3: READMEs

A README is the first 5 minutes of a new teammate's experience. **Hand-curate**. Never autogenerate.

Required sections:

```markdown
# Project name

One-line elevator pitch.

## What this is

2-3 sentences. The problem this solves and who for.

## Quickstart

```bash
git clone ...
cd ...
make setup        # installs deps, creates .env, seeds DB
make test         # runs the test suite
make dev          # starts the dev server on port 3000
```

## Stack

Bullets: language, framework, DB, deployment. Link to relevant ADRs for the why.

## Layout

```
src/                source
tests/              tests (unit + integration, separated by markers)
docs/adr/           architecture decisions
docs/runbook/       operational playbooks
```

## Conventions

Link to CLAUDE.md (which has the AI-agent conventions). For humans:

- How to run tests
- How to deploy
- Where to file bugs / feature requests

## See also

- `docs/adr/` — every architectural decision
- `docs/runbook/` — operational playbooks
```

**Anti-patterns**:
- Auto-generated tables of contents from headings (rarely useful, often stale).
- Marketing language ("blazing fast", "revolutionary").
- Stale install instructions (test them when you onboard the next teammate).
- 500 lines that nobody reads. Trim aggressively — link to deeper docs for details.

## Anti-rationalization table

| You're thinking | The reality |
|-----------------|-------------|
| "I'll write the ADR after we ship." | The decision context evaporates within a week. Write it as part of the PR that lands the decision. |
| "The git history is the docs." | `git log` shows *what* changed, never *why*. Future-you will not run `git blame` on every line of a 50-file PR to reconstruct your reasoning. |
| "The code is self-documenting." | Names show intent within a function. They do not show why this function exists, why it returns what it returns, or which ADR mandated its shape. |
| "Nobody reads ADRs." | They do, when something breaks. The ADR that prevents one wasted "should we rewrite this?" debate has paid for itself. |
| "Docs go stale, so why bother?" | Docs go stale because they aren't versioned with code. ADRs are immutable + numbered, so they don't go stale — they get superseded. That's a different failure mode and a much smaller one. |

## Quick checklist before merging

```
[ ] If this PR makes an architectural decision, an ADR is in the same PR
[ ] Every public function/route has a docstring that explains WHY (not what)
[ ] If a public API changed, the README's Quickstart still works
[ ] If a major dep was added, an ADR exists explaining why
[ ] If the layout changed, the README's "Layout" section is updated
```
